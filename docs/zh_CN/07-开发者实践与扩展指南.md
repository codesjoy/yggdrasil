# 07. 开发者实践与扩展指南


> 本文为应用开发者、模块作者、provider 作者、配置作者和运维人员提供实践模板与 checklist。
>
> 关键词：App、Hub、Module、Capability、assembly.Spec、prepared runtime assembly、Prepare、Compose、BusinessBundle、Staged Reload。


## 1. 业务应用接入模板

### 1.1 推荐 main.go

```go
func main() {
    if err := yggdrasil.Run(
        context.Background(),
        business.Compose,
        yggdrasil.WithConfigPath("app.yaml"),
    ); err != nil {
        panic(err)
    }
}
```

`yggdrasil.Run` 面向单主 App 的 main 程序，可安装进程默认 logger / OTel / legacy instance facade。测试、多 App、sidecar 或嵌入式场景应使用 `yggdrasil.New` / `app.New`，并保持默认的 App-local runtime；只有确实需要兼容全局 API 时才显式设置 `WithProcessDefaults(true)`。

### 1.2 推荐 business.Compose

```go
func Compose(rt yggdrasil.Runtime) (*yggdrasil.BusinessBundle, error) {
    userClient, err := rt.NewClient(context.Background(), "user-service")
    if err != nil { return nil, err }

    svc := &OrderService{
        Users:  userClient,
        Logger: rt.Logger(),
    }

    return &yggdrasil.BusinessBundle{
        RPCBindings: []yggdrasil.RPCBinding{{
            ServiceName: "OrderService",
            Desc:        orderpb.OrderServiceDesc,
            Impl:        svc,
        }},
    }, nil
}
```

## 2. 模块作者 Checklist

### 2.1 基础模块

- [ ] `Name()` 全局唯一且稳定。
- [ ] 强依赖用 `DependsOn()`，不要用 `InitOrder()` 表达。
- [ ] 需要配置时实现 `Configurable`。
- [ ] `Init()` 只初始化长期资源，不对外 serve。
- [ ] `Start()` 启动后台过程或服务行为。
- [ ] `Stop()` 幂等，使用 `sync.Once` 或原子状态。
- [ ] 可热更新时实现 `Reloadable`，并保证 prepare/commit/rollback 语义。
- [ ] 暴露 capability 时声明 `CapabilitySpec` 的 name、cardinality、type。
- [ ] 需要进程全局默认值时实现 `IsolationReporter` 并声明 `IsolationModeRequiresProcessDefaults`。

### 2.2 参与自动装配的模块

- [ ] 实现 `AutoDescribed`。
- [ ] `AutoRule.Match()` 是纯函数。
- [ ] 声明 `AffectedPaths()`。
- [ ] `DefaultPolicy` 分数清晰，避免与同类模块同分冲突。
- [ ] 相关决策能在 `assembly.Spec.Decisions` 中解释。

## 3. Provider 作者 Checklist

- [ ] Provider 本身可以进 Hub，但运行时动态实例不能进 Hub。
- [ ] Prepare 阶段只构造对象和内部 helper。
- [ ] Start 阶段才 bind/listen/register。
- [ ] Close/Stop 路径完整，失败时不泄漏 goroutine。
- [ ] 若无法二阶段拆分，不满足 Yggdrasil 接入契约。

### 3.1 何时用 CapabilityRegistration，何时用 Module

- 只想给现有 capability 增加 provider，且不需要 `Start/Stop/Reload/DependsOn` 时，优先用 `WithCapabilityRegistrations(...)`。
- 需要生命周期、依赖 DAG、热更新或自动装配时，继续使用 `WithModules(...)` 和完整 `module.Module`。

`CapabilityRegistration` 适合 transport provider、logger handler、resolver/balancer、REST middleware 这类 provider-only 扩展。

约束：

- `Capabilities` 回调可能早于 `Init` 执行，且可能被多次调用。
- `Capabilities` 必须是纯函数、确定性、无副作用。
- 若 capability value 依赖配置，必须返回惰性 provider/factory，在真正构造 runtime 对象时再读取共享状态。
- `CapabilityRegistration` 不支持 `Reloadable`；配置变更需要自动重建 provider 时，应改用完整模块。

## 4. 配置作者 Checklist

- [ ] 使用 `yggdrasil.mode` 表达环境/组合模式。
- [ ] 默认能力不符合预期时，优先用显式配置或 `force_defaults`。
- [ ] 链式扩展优先使用版本化模板。
- [ ] 高风险组件如 auth/retry/circuit-breaker 显式配置，不依赖默认模板。
- [ ] 出现 AmbiguousDefault 时，不要调整模块注册顺序；应显式指定 provider 或禁用候选。

## 5. 运维排障 Checklist

### 5.1 启动失败

1. 看失败阶段：Plan / Seal / Init / Prepare / Compose / Install / Start。
2. Plan 失败：看 mode、template、default selector、override。
3. Seal 失败：看 DAG、缺失依赖、capability cardinality。
4. Init 失败：看模块配置 decode 与资源初始化。
5. Compose 失败：看业务 client、业务依赖构造。
6. Install 失败：看服务名、desc/impl 类型、route 冲突。
7. Start 失败：看 transport bind、registry、server handler。

### 5.2 Reload 异常

1. 先查看 old/new plan diff。
2. 若模块集合变化，通常需要 restart-required。
3. 若 business bundle 结构变化，首版需要 restart-required。
4. 若 PrepareReload 失败，检查新配置合法性。
5. 若 Commit 失败，检查模块内部提交逻辑与外部依赖。
6. 若 Rollback 失败，查看 degraded 状态并安排重启。

## 6. 常见反模式

| 反模式 | 问题 | 推荐做法 |
|---|---|---|
| 在 package init 中注册全局 provider | 破坏 App 实例隔离 | 显式模块注册或 auto assembly |
| 模块运行期读取 slog/OTel/instance 全局状态 | 依赖 process-default facade，多个 App 时串台 | 通过 Runtime、Capability 或构造参数接收 App-local 依赖 |
| 用 InitOrder 表达强依赖 | 同层 tie-breaker 不能保证依赖 | 使用 DependsOn |
| Capability 冲突时取第一个 | 非确定性且不可诊断 | 使用基数校验和显式配置 |
| Compose 直接操作 server runtime | 绕过安装边界 | 返回 BusinessBundle |
| Prepare 阶段监听端口 | 破坏 runtime-ready 契约 | Start 阶段 bind/listen |
| Reload 自动重跑业务 Compose | 首版复杂度过高 | 标记 restart-required |
| 把 resolver watch 放进 Hub | 高频动态对象边界错误 | 由 client runtime 管理 |

## 7. 推荐错误信息格式

错误信息建议包含：

- stage：Plan / Seal / Init / Prepare / Compose / Install / Start / Reload；
- target：module / capability / template / binding；
- reason：失败原因；
- available：可选候选；
- fix：修复建议。

示例：

```text
stage=Plan target=capability:observability.logger.handler reason=AmbiguousDefault
candidates=json-handler,text-handler
fix=configure yggdrasil.overrides.force_defaults.observability.logger.handler or disable one candidate module
```

## 8. 版本化文档维护建议

- 架构与 Hub 文档偏稳定，作为长期设计基线。
- Bootstrap 文档与 proposal 版本绑定，保留 changelog。
- Lifecycle 文档应随 public API 变更同步更新。
- Config/Reload 文档应包含最新错误语义和 restart-required 规则。
- Transport 文档应随协议 provider、新安全 profile、新 balancer 同步扩展。
