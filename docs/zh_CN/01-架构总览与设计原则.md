# 01. Yggdrasil v3 架构总览与设计原则


> 本文是 Yggdrasil v3 文档集的一部分，面向开发者、模块作者和维护者，总结框架架构、设计意图和边界。
>
> 关键词：App、Hub、Module、Capability、assembly.Spec、prepared runtime assembly、Prepare、Compose、BusinessBundle、Staged Reload。


## 1. 框架定位

Yggdrasil 是一个基于 Go 语言开发的微服务框架库。默认应用路径围绕 `App` 实例和 Module Hub 组织，而不是进程级业务组合单例；它也不是一个强绑定特定 DI 容器的应用框架。内核使用 App-local runtime，进程默认 logger、OTel provider、legacy instance facade 只是可选兼容入口，不能反向污染核心路径。它的核心定位是：

- 为 RPC / REST 微服务提供稳定运行时；
- 用统一模块中心管理日志、传输、注册发现、负载均衡、可观测性等基础设施能力；
- 通过声明式装配系统把配置、模块选择、默认实现和扩展链模板编译成可审计的计划；
- 在 runtime-ready 后给业务一个正式组合阶段，让业务安全地构造依赖下游 client 的服务实现。

## 2. 总体架构

```text
Business Application
  |
  v
Bootstrap / Planning Layer
  - Config Load
  - Mode Resolve
  - Auto Module Planning
  - Default Selection
  - Chain Template Expansion
  - assembly.Spec
  |
  v
App / Hub Layer
  - Hub.Use / Seal / Init / Start / Stop / Reload
  - Capability Resolution
  - prepared runtime assembly
  |
  v
Business Compose / Install Layer
  - Runtime narrow surface
  - Compose(Runtime) -> BusinessBundle
  - InstallBusiness
  |
  v
Serve / Wait / Stop
```

这条路径体现了两个核心分离：

1. **计划与实例分离**：`assembly.Spec` 是当前公共声明式计划，参与 explain / diff / hash；prepared runtime assembly 是内部实例图，只存在于进程内。
2. **框架内核与业务图分离**：模块和 provider 进入 Hub；业务服务实现、handler、task、hook 进入 BusinessBundle，不进入 Hub。
3. **App-local 与 process-default 分离**：`app.New` / `yggdrasil.New` 默认不改写进程全局状态；`yggdrasil.Run` 面向单主 App main 程序，可安装进程默认兼容 facade。同一时刻最多一个 App 持有 process defaults。

## 3. 关键分层

| 层次 | 职责 | 典型对象 |
|---|---|---|
| 入口层 | 创建 App、读取配置入口、暴露高层 API | `yggdrasil.Run`, `yggdrasil.New`, `app.New` |
| 规划层 | 解析 mode、选择模块、选择默认实现、展开链模板 | `assembly.Spec`, `assembly.Decision` |
| 容器层 | 注册模块、DAG 校验、Capability 校验、生命周期协调 | `module.Hub` |
| Runtime 准备层 | 构造 client manager、server object、resolver/balancer runtime 等 | prepared runtime assembly, `Runtime` |
| 业务组合层 | 构造业务 impl、注册 RPC/REST/HTTP/task/hook | `BusinessBundle` |
| 运行层 | 对外监听、实例注册、优雅关闭、热重载 | `Start`, `Wait`, `Stop`, `Reload` |

## 4. 设计原则

### 4.1 核心只定义组织结构，不定义业务能力枚举

内核只提供 `App`、`Hub`、`Module`、生命周期接口、依赖声明接口和 capability 解析工具。日志、传输、注册中心、resolver、balancer、interceptor 等能力由各自子系统定义 capability spec 与 provider 接口。

### 4.2 模块依赖顺序与调用链顺序分离

- 模块初始化、启动、停止顺序来自显式依赖 DAG。
- RPC interceptor 与 REST middleware 的执行顺序来自配置中的有序列表或展开后的链模板。
- `InitOrder()` 只能作为同层 tie-breaker，不得表达强依赖。

### 4.3 隐式装配必须产出显式计划

任何自动选择都必须写入 `assembly.Spec`，包括：启用哪些模块、默认实现如何选中、链模板如何展开、决策来自配置还是 mode。这样 dry-run、explain、diff 和 reload 才有稳定真相源。

### 4.4 Hub 管理长期模块，不管理高频动态对象

Hub 适合管理 logger provider、transport provider、registry provider、interceptor provider 等长期对象。不适合管理 per-request、per-stream、endpoint connection、picker、resolver watch 等高频动态实例。

### 4.5 失败语义必须明确

- 启动失败必须逆序补偿停止已启动模块。
- Stop 必须幂等。
- Reload 采用 staged reload，不允许隐式部分提交。
- Capability 冲突必须报错，不允许“取第一个”。
- Rollback 失败必须进入 degraded / restart-required，而不是无限重试。

## 5. 包结构建议

```text
yggdrasil/
  app/                  # App 生命周期、业务安装、reload、内部 facade
  assembly/             # Plan / DryRun / Spec / Diff / Hash / Explain
  module/               # Hub、Module、DAG、Capability、Lifecycle、Scope
  config/               # Manager、Snapshot、View、Source
  transport/            # 协议无关 transport contract 与 provider
  transport/protocol/   # gRPC、HTTP 等协议实现
  transport/runtime/    # client/server runtime orchestration、balancing
  discovery/            # registry / resolver contract
  observability/        # logger、otel、stats
  admin/                # governor / diagnostics
```

## 6. 稳定 API 与 internal 边界

`app` 包保持稳定的 public API 和所有 `func (a *App)` 方法；实现细节放到 `app/internal/*`：

```text
app
  -> app/internal/assembly
  -> app/internal/bootstrap
  -> app/internal/install
  -> app/internal/lifecycle
  -> app/internal/runtime
```

这能让 public API 稳定，同时允许内部实现持续演进。

## 7. 架构收益

- 默认业务接入路径更简单：`yggdrasil.Run -> business.Compose -> Wait`；`app.New -> Prepare -> ComposeAndInstall -> Start -> Wait` 保留给高级控制场景。
- 框架内核仍严格：DAG、Capability 基数、Scope、Reload 状态都可校验。
- 默认行为可审计：每个默认选择和模板展开都能 explain。
- Reload 更可靠：比较对象是稳定的 `assembly.Spec`，不是运行时实例。
- 扩展更安全：新增协议、注册中心、resolver、middleware 时不需要改核心枚举。
